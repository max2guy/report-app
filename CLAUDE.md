# Project Overview
- 이 프로젝트는 작은 단위로 안전하게 수정한다.
- 기존 구조와 동작을 최대한 유지한다.
- 필요한 부분만 최소 수정하는 것을 우선한다.
- 나는 초보 개발자이므로, 변경 이유를 짧고 분명하게 설명한다.

# Communication Style
- 작업 전에 무엇을 바꾸려는지 짧게 설명한다.
- 여러 방법이 있으면 가장 단순하고 안전한 방법을 먼저 제안한다.
- 어려운 용어를 쓰면 짧게 뜻을 설명한다.
- 확실하지 않은 내용은 추측하지 말고 불확실하다고 말한다.
- 에러가 나면 해결책만이 아니라 원인도 함께 설명한다.

# Package Manager
- 패키지 매니저는 pnpm을 우선 사용한다.
- 기존 프로젝트가 npm 또는 yarn을 명확히 사용 중인 경우가 아니면 바꾸지 않는다.

# Commands
- install: pnpm install
- dev: pnpm dev
- build: pnpm build
- lint: pnpm lint
- typecheck: pnpm typecheck
- test: pnpm test

# General Rules
- 변경 범위는 최소화한다.
- 관련 없는 파일은 건드리지 않는다.
- 승인 없이 대규모 리팩터링하지 않는다.
- 승인 없이 의존성을 대량 추가하거나 교체하지 않는다.
- 기존 코드 스타일이 있으면 먼저 따른다.
- 주석은 꼭 필요한 곳에만 추가한다.
- 복잡한 패턴보다 이해하기 쉬운 코드를 우선한다.

# Coding Preferences
- 가능하면 TypeScript를 우선 사용한다.
- any 사용은 꼭 필요한 경우에만 한다.
- 함수와 변수 이름은 역할이 드러나게 작성한다.
- 긴 함수는 가능하면 더 작은 함수로 나눈다.
- 하드코딩보다 기존 상수, 유틸, 패턴을 우선 재사용한다.

# Workflow
- 코드를 수정하기 전에 관련 파일을 먼저 읽고 구조를 파악한다.
- 가장 작은 수정으로 먼저 해결을 시도한다.
- 수정 후 가능한 범위에서 lint, typecheck, test를 확인한다.
- 실행하지 못한 검증은 무엇을 못 했는지 분명히 말한다.

# File Editing Rules
- 기존 파일 포맷을 유지한다.
- 관련 없는 포맷팅 변경은 하지 않는다.
- import 정리만을 위한 불필요한 수정은 하지 않는다.
- 한글 UI 문구가 있으면 기존 톤을 유지한다.

# Debugging Rules
- 문제가 생기면 증상, 원인 후보, 가장 가능성 높은 원인, 수정 방향 순서로 설명한다.
- 한 번에 여러 군데를 바꾸기보다 원인 가능성이 큰 부분부터 수정한다.
- 재현 방법이 보이면 먼저 짧게 정리한다.

# Beginner-Friendly Output
- 수정한 파일과 변경 이유를 짧게 요약한다.
- 명령어는 바로 복붙 가능한 형태로 제시한다.
- 선택지가 여러 개면 추천안을 먼저 말하고 대안은 짧게 덧붙인다.

# Safety / Change Control
- 환경변수, 인증, 배포 설정, DB 관련 변경은 특히 신중하게 다룬다.
- 데이터 손실 가능성이 있는 작업은 먼저 경고한다.
- 위험한 삭제, 초기화, 대량 이동은 꼭 이유와 영향을 설명한다.

# Do Not
- 관련 없는 버그를 한 번에 고치려 하지 말 것
- 승인 없이 파일 구조를 크게 바꾸지 말 것
- 승인 없이 패키지를 무분별하게 추가하지 말 것
- 추측을 사실처럼 말하지 말 것
- 문제를 숨기지 말 것
